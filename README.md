# 마지막 페이지 (The Last Page)

독서 후 개인적인 감상과 기록을 체계적으로 관리할 수 있는 iOS 독서 기록 앱입니다.<br>
AI기반 추천 키워드를 통해 글의 소재를 추천받을 수 있으며 사용자의 감정에 따라 
배경색이 다르게 적용되어 보다 깊고 다양한 기록을 할 수 있습니다.

<br>

## 주요 구현 내용

### 1. Clean Architecture + MVVM 구조

#### 아키텍처 개요

이 프로젝트는 Clean Architecture와 MVVM 패턴을 결합한 구조로 설계되었습니다. 각 레이어가 분리되어 의존성 역전 원칙을 따르고 있습니다.

---

#### 1. Domain Layer (도메인 레이어)


- 비즈니스 로직의 핵심: 애플리케이션의 핵심 비즈니스 규칙과 엔티티를 정의

- 외부 의존성 없음: 다른 레이어에 의존하지 않는 순수한 비즈니스 로직

- 인터페이스 정의: 저장소와 유스케이스의 프로토콜을 정의하여 의존성 역전 원칙 구현

#### 주요 구성 요소

**엔티티 (Entities)**

- 도메인의 핵심 데이터 모델들

- 순수한 Swift 구조체로 구현되어 외부 의존성 없음

- 비즈니스 규칙을 캡슐화

**유스케이스 (UseCases)**

- 데이터베이스 관련 유스케이스들: 도서 조회, 저장, 수정, 삭제

- 네트워크 관련 유스케이스들: 도서 검색, 키워드 조회, 배경색 조회

- 각각 특정 비즈니스 기능을 캡슐화

**저장소 프로토콜 (Repository Protocols)**

- 데이터 접근 인터페이스 정의

- 구현체는 데이터 레이어에서 제공

---

#### 2. Presentation Layer (프레젠테이션 레이어)

- UI 로직: 사용자 인터페이스와 상호작용 처리

- 상태 관리: 뷰모델을 통한 UI 상태 관리

- 네비게이션: 코디네이터 패턴을 통한 화면 전환 관리

---

#### 3. Data Layer (데이터 레이어)

- 데이터 접근: 로컬 데이터베이스와 원격 API 접근

- 데이터 변환: 도메인 엔티티와 데이터 모델 간의 매핑

- 외부 시스템 연동: 데이터베이스, 네트워크 API 등과의 통신

##### 주요 구성 요소

로컬 데이터 (Local Data)

- 로컬 데이터베이스 접근 구현

- 실제 데이터베이스 작업 수행

- 데이터베이스 모델과 도메인 엔티티 간 변환

원격 데이터 (Remote Data)

- 네트워크 API 접근 구현

- 실제 HTTP 요청 처리

- API 응답 데이터 처리

데이터 모델 (Models)

- 데이터베이스 모델

- API 응답 데이터 모델

---

#### 데이터 흐름

##### 1. 사용자 액션 → 데이터 요청 흐름

> 사용자 액션 → 뷰 컨트롤러 → 뷰모델 → 유스케이스 → 저장소 → 데이터소스/네트워크매니저


##### 2. 데이터 응답 → UI 업데이트 흐름

>데이터소스/네트워크매니저 → 저장소 → 유스케이스 → 뷰모델 → 뷰 컨트롤러 → UI 업데이트

##### 3. 의존성 주입 흐름

>의존성 주입 컨테이너 → 로컬/원격 컨테이너 → 유스케이스 → 뷰모델 → 뷰 컨트롤러

### 2. Coordinator Pattern을 통한 화면 전환 아키텍처

복잡한 중첩 화면 전환 문제를 해결하기 위해 계층적 Coordinator 구조를 설계했습니다. HomeCoordinator → ReadingCoordinator → EditInfoCoordinator와 같은 다층 네비게이션에서 각 화면의 생명주기를 명확하게 관리할 필요가 있었습니다.

```swift
protocol Coordinator: AnyObject {
    var childCoordinators: [Coordinator] {get set}
}
```

MainCoordinator가 TabBar 기반의 4개 주요 탭을 관리하며, 각 탭별로 독립적인 NavigationController를 할당했습니다. 화면이 닫힐 때 자식 Coordinator를 자동으로 제거하는 메모리 관리 로직을 구현하여 메모리 누수를 방지했습니다.

동일한 화면에서 상황에 따라 push, present, 커스텀 프레젠테이션 등 다양한 전환 방식이 필요한 경우, ViewController는 "showCategories"처럼 의미적 호출만 하고 실제 전환 방식은 Coordinator에서 결정하도록 분리했습니다.

### 3. Combine 기반 반응형 데이터 흐름

독서 기록 앱의 특성상 책 추가/삭제 이벤트가 홈, 아카이브, 통계 화면에 동시에 영향을 미치는 복잡한 데이터 동기화가 필요했습니다. 모든 데이터 작업을 Publisher-Subscriber 패턴으로 구현하여 비동기 처리와 실시간 UI 동기화를 하였습니다.

검색 기능에서는 사용자 입력의 효율적 처리를 위해 시간 기반 연산자를 활용했습니다:

```swift
input.query
    .debounce(for: .milliseconds(300), scheduler: DispatchQueue.main)
    .removeDuplicates()
    .sink { [weak self] query in
        // 실제 검색 로직 수행
    }
    .store(in: &cancellables)
```

Archive 화면에서는 검색어, 상태 태그, 카테고리 태그, 감정 태그의 네 가지 독립적인 필터 조건을 실시간으로 조합하기 위해 Publishers.CombineLatest를 사용했습니다.
모든 ViewModel에서 `Set<AnyCancellable>`과 weak self 패턴을 일관되게 적용하여 순환 참조를 방지하고, receive(on: DispatchQueue.main)을 배치하여 UI 업데이트의 스레드 안전성을 보장했습니다.

## 핵심 기능 설명

### 1. VisionKit과 NaturalLanguage를 통한 문서 스캔 및 텍스트 추출

사용자가 복잡한 입력 과정 없이 책의 내용을 쉽게 메모로 옮길 수 있도록, 카메라로 텍스트를 스캔하고 원하는 문장을 선택해 추가하는 기능을 구현했습니다.

- **기술** : `VisionKit`, `VNDocumentCameraViewController`, `VNRecognizeTextRequest`, `NaturalLanguage`,`NLTokenizer`

1. **텍스트 추출**
- 카메라 화면을 띄워 사용자가 책 페이지를 촬영하면, 그 이미지를 분석해 텍스트를 추출합니다.

2. **문장 단위 분리**
- 사용자가 원하는 부분만 쉽게 골라 사용할 수 있도록 텍스트를 문장 단위로 나누어 보여줍니다.
- 문장이 적거나 형식이 다양할 경우, 줄바꿈을 기준으로 나누는 로직을 적용했습니다.
- 분리된 문장들은 화면에 테이블뷰 형태로 표시됩니다. 사용자는 이 목록에서 원하는 문장을 선택할 수 있으며, 선택한 문장은 현재 메모 작성 중인 위치에 바로 삽입됩니다.

### 2. AI 기반 감정 색상 시스템

감정적 경험이 기억에 오래 남는다는 점에 착안하여, 사용자가 책으로부터 최대한 많은 경험과 학습, 여운을 가져가도록 하는 것이 목표였습니다. 이를 위해 사용자가 책을 읽으며 느낀 감정을 자세한 감정 키워드를 통해 쉽게 떠올리도록 하고, 해당 키워드에 맞는 색상을 배경색으로 지정해 더 직관적이고 기억에 남는 경험을 제공하고자 했습니다.

복잡한 인간의 감정을 가능한 적절한 색상으로 표현하기 위해  ChatGPT API에 최적화된 프롬프트를 설계하여 감정 태그들을 분석하고 그에 맞는 색상을 선정하도록 했습니다. 특히 복합적인 감정을 표현하기 위해 그라데이션을 활용하기로 결정했으며, 그라데이션의 시작점과 끝점 좌표까지 정확히 저장하여 AI가 의도한 색상 변화 방향을 보존해야 했습니다.

##### 데이터 구조 설계
이중 데이터 구조를 사용하여 각 레이어에서 효율적으로 데이터를 처리할 수 있도록 설계했습니다.:

- **애플리케이션 레벨**: `BackColorEntity`로 색상 배열과 함께 `startPoint`, `endPoint`를 `CGPoint` 타입으로 보관하여 UI에서 바로 사용할 수 있도록 했습니다.

- **데이터베이스 레벨**: `BackColorObject` 클래스로 동일한 정보를 저장하되, `CGPoint`를 `startX`, `startY`, `endX`, `endY`로 분해하여 `Double` 타입으로 저장했습니다. 이때 `EmbeddedObject`를 활용해 구조화된 데이터 관리를 구현했습니다.

##### 최적화 전략
API 호출 비용을 최소화하기 위해 다음 조건을 모두 만족할 때만 API를 호출하도록 했습니다:
- 기존 배경색이 없는 경우
- 새로운 감정 태그가 존재하는 경우
- 기존 감정 태그 구성과 실제로 다른 경우

### 3. 사용자 친화적 키보드 인터페이스 구현
독서 메모 앱의 특성상 장문의 텍스트 작성이 필요했습니다.
이에 따라 텍스트 작성 시, 키보드가 커서를 가리는 문제를 해결하기 위해 텍스트뷰의 커서 위치를 실시간으로 추적하는 시스템을 구현했습니다.
- **실시간 커서 추적 시스템**: 키보드 높이와 텍스트뷰 내의 커서의 절대 좌표를 계산하여, 커서가 키보드 영역에 가려지지않도록 자동 스크롤을 적용했습니다.
- **동적 인터페이스 조정**: 사용자가 입력하는 텍스트 분량에 따라 텍스트의 높이를 동적으로 조정하여 인터페이스가 자동으로 확장되도록 하였습니다.

### 4. 에러 처리

이 프로젝트는 네트워크, 데이터베이스, 파일 시스템, 권한 관련 에러를 체계적으로 처리하여 안정적이고 사용자 친화적인 애플리케이션 경험을 제공합니다. 에러 처리는 재시도 로직, 사용자 피드백, 시스템 복구, 로깅으로 구성되며, 각 모듈별로 다음과 같이 설계되었습니다.

#### 1. 네트워크 에러 처리

네트워크 에러는 HTTP 상태 코드(400, 401, 403, 404, 429, 500/503), 연결 문제(타임아웃, 연결 없음, 잘못된 응답 형식), 데이터 처리 에러(JSON 파싱 실패, 필수 필드 누락 등)로 분류됩니다. 처리 방식은 다음과 같습니다:

- **자동 재시도**: 서버 오류, 타임아웃, 요청 한도 초과 등 재시도 가능한 에러에 대해 최대 3회, 지수 백오프(2초, 4초, 6초)로 재시도합니다. 잘못된 요청(400)이나 인증 실패(401) 등은 재시도하지 않습니다. 
- **네트워크 모니터링**: NetworkMonitor로 실시간 연결 상태를 확인하고, 요청 전 연결 검증으로 불필요한 요청을 방지합니다. 
- **사용자 피드백**: 재시도 가능한 에러는 "재시도" 버튼과 함께 Alert로 안내하고, 불가능한 에러는 Toast 메시지로 표시합니다. 
- **데이터 정제**: 알라딘 API의 비정상 제어 문자를 제거하고, 손상된 JSON 구조를 복구하며, 상세한 에러 로깅(발생 시점, 엔드포인트, 재시도 횟수)을 수행합니다.

#### 2. 데이터베이스 에러 처리

데이터베이스 에러는 Realm 관련 문제(초기화 실패, 저장/조회 실패, 마이그레이션 실패, 파일 손상, 디스크 공간 부족, 동시성 충돌 등)로 나뉩니다. 처리 방식은 다음과 같습니다:

- **에러 통일**: 모든 에러를 `DatabaseError` enum으로 관리하며, 사용자 메시지와 재시도 가능 여부를 포함합니다. 
- **사용자 피드백**: 재시도 가능한 에러(일시적 저장 실패 등)는 Alert로 재시도 옵션을 제공하고, 마이그레이션 실패나 파일 손상은 Toast로 안내합니다. 
- **시스템 복구**: 마이그레이션 실패 시 앱 재시작 또는 데이터 초기화를 유도하고, 파일 손상 시 백업 복구를 시도합니다. 디스크 공간 부족 시 작업을 중단하고 경고를 표시합니다. 
- **스레드 안전성**: 전용 큐에서 Realm 작업을 수행하고, 쓰기 작업은 트랜잭션 내에서만 실행합니다.

#### 3. 파일 시스템 에러 처리

파일 시스템 에러는 저장 공간(디스크 용량 부족, 한도 초과), 권한, 이미지 처리, 파일 상태, 네트워크 관련 에러로 구분됩니다. 처리 흐름은 다음과 같습니다:

- **에러 분류 및 재시도**: 디스크 용량 부족, 권한 문제는 재시도 불가능하며, 파일 사용 중이나 이미지 처리 실패는 재시도 가능합니다. 
- **앱 시작 시**: 최소 10MB 여유 공간을 확인하고, 부족 시 30일 이상된 파일을 자동 삭제합니다. 
- **이미지 처리**: 저장 시 1024x1024로 리사이즈하고, 다중 압축(0.8→0.2 품질)을 시도하며, 로드 시 경로와 권한을 검증합니다. 
- **사용자 안내**: 재시도 가능한 에러는 Alert, 불가능한 에러는 Toast로 표시하며, 해결 방법이 있는 경우 도움말 버튼을 제공합니다.

#### 4. 권한 관련 에러 처리

카메라, 사진 라이브러리, 네트워크, 알림 권한 관련 에러를 처리하며, 사용자 중심 원칙을 따릅니다:

- **사전 확인 및 점진적 요청**: 기능 사용 전 권한 상태를 확인하고, 필요 시에만 요청하여 사용자 경험을 최적화합니다. 
- **대안 제공**: 권한 거부 시 오프라인 모드나 대체 기능을 제공하고, 설정 화면 이동 경로를 안내합니다. 
- **실시간 모니터링**: 앱 복귀 시 권한 상태를 재확인하고, 네트워크 변화에 대응합니다.

#### 5. 에러 로깅 및 모니터링

모든 에러는 타임스탬프, 컨텍스트, 심각도, 재시도 여부, 해결 방법을 포함해 로깅됩니다. 저장 공간은 지속적으로 모니터링하며, 30일 이상된 파일은 자동 삭제됩니다. 복잡한 문제에는 단계별 가이드를 제공하고, 설정 앱 연동으로 문제를 해결하도록 유도합니다.

이 설계는 안정성과 사용자 경험을 최우선으로 하여, 다양한 에러 시나리오에 대해 체계적이고 투명한 처리 방식을 제공합니다.

### 5. 유연하고 일관된 시스템으로 위젯을 구현했습니다.

**메모리 효율성을 고려한 타임라인 아키텍처를 구성했습니다.**
단일 엔트리 배치 처리를 구현하여 30초 업데이트 주기 내에서 모든 데이터 변경사항을 하나의 타임라인으로 관리하여 메모리 사용량을 줄였습니다.  
  
**AppIntent를 활용해 위젯 타입을 동적으로 전환했습니다.**
사용자가 위젯에서 직접 타입을 변경할 수 있도록 AppIntent를 활용했습니다. iOS17 이상에서는 위젯 내부에서 통계 위젯과 랜덤 도서 위젯을 동적으로 전환할 수  
있고, iOS16에서는 기본 통계 위젯을 제공합니다.  
  
**모든 위젯을 단일 JSON 구조로 통합했습니다.**
전체 위젯 데이터를 하나의 JSON 객체로 관리하여 데이터가 일관되도록 하였습니다. 메인 앱에서 데이터 변경 시`WidgetCenter.shared.reloadTimelines`를  
실행하여 즉시 위젯에 반영됩니다.  
  
**공통 모디파이어를 통해 위젯 코드를 일관되게 유지했습니다.**
재사용 가능한 CustomModifier를 구현하여 iOS 버전별 배경 처리를 일관되게 관리했습니다. 위젯 패밀리별 레이아웃 분기도 모듈화하여 유지보수성을 높였습니다.
